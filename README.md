# astrodar

`astrodar` 是一个本地计算、交互式终端天文历程序。它从时间尺度、解析轨道理论和坐标变换出发计算天文与历法数据，而不是查询在线历表。

当前程序聚焦于太阳、月球、节气、八字年/月/日/时柱，以及柏林和北京两个独立地区的本地显示。

## 平台与依赖

- macOS 与 Linux；需要 POSIX 时间接口和 IANA 时区数据库。
- C++17 编译器。
- [FTXUI](https://github.com/ArthurSonzogni/FTXUI) 开发库，用于终端界面。

macOS/Homebrew：

```sh
brew install ftxui
make
make run
```

Linux 请先通过发行版软件包或 FTXUI 源码安装 FTXUI，再运行：

```sh
make
make run
```

默认 `Makefile` 会在 macOS 上通过 Homebrew 自动定位 FTXUI；若库安装在非标准目录，可以指定：

```sh
make FTXUI_PREFIX=/path/to/ftxui
```

### 地点配置

地点面板分为两种互斥类型：

- **时区**：仅保存 IANA 名称，例如 `Europe/Berlin`。程序从系统 IANA `zone1970.tab` 或 `zone.tab` 读取代表性坐标；日出日落等数据标为该时区的参考值。
- **坐标**：保存用户指定的地点名称、纬度和经度，不绑定法定时区；时间与太阳活动按真太阳时计算。

配置文件为 `locations.json`。Linux 路径是 `$XDG_CONFIG_HOME/astrodar/locations.json`（未设置时为 `~/.config/astrodar/locations.json`）；macOS 路径是 `~/Library/Application Support/astrodar/locations.json`。文件只保存原始选择，不保存从 IANA 数据库派生的参考坐标，并采用临时文件加原子重命名写入。

按 `A` 打开“添加地点”。确认时会校验输入、立即加入概览并写入配置；概览最多保留四个地点面板。按 `E` 打开编辑页：上下选择地点，`Enter` 选中后可用上下调整顺序，退格删除并经确认后写入配置。

### 概览页

- BER 与 BEI 两个独立时区卡片；每张卡片按自身时区显示本地时间、刻和八字年/月/日/时柱。
- 时区卡片显示来自系统 IANA 数据库的参考坐标；日出日落等太阳活动是该参考坐标的结果，而非整个时区的统一值。
- 概览最多显示四个地点，按两列两行排列；页面按实际内容高度布局，不会将地点卡片纵向拉伸至终端底部。
- 太阳事件区按地点坐标显示日出、太阳中天、日落，以及民用、航海、天文晨昏蒙影的本地时间。
- 矮终端（低于 32 行）会自动将太阳活动时间线折叠为两列；高度充足时维持完整单列时间线。

### 天文数据页

- 上方：UTC、Unix 时间戳、JD、MJD、JD(TT)。
- 中间左栏：太阳和月球的文字数据块，分别显示位置、距离和照明等量。
- 中间右栏：连续计算的月相盘，使用 `░` 和 `▓` 绘制暗面与亮面；圆盘先在逻辑坐标中计算，再在输出阶段以横向字符倍率做栅格补偿。
- 下方：当前/下一节气与本年八个主节气的 UTC 时刻。

显示层通过 [localization.cpp](localization.cpp) 集中管理中文文案。当前仅提供中文 UI；标准缩写如 UTC、TT、JD、MJD、AU 与 km 保留。

### 坐标查询弹窗

按 `F` 打开覆盖式坐标查询。查询不使用法定时区，而是以经度和均时差计算真太阳时；适用于任意经纬度地点的时间、干支与太阳活动查询。

| 按键 | 功能 |
| --- | --- |
| `↑` / `↓` | 选择输入行；在自定义 UTC 编辑中调整当前时间段 |
| `Tab` | 切换当前行的模式：时间来源、纬度 N/S 或经度 E/W |
| `Enter`（自定义 UTC） | 进入或退出分段时间编辑；`←` / `→` 选择年、月、日、时、分、秒 |
| 数字、小数点、度分秒符号、退格 | 编辑 JD、纬度或经度字段 |
| `F` / `Esc` | 关闭查询弹窗 |

时间来源可选实时 UTC、自定义 UTC 与 `JD(TT)`。实时 UTC 的时间行只读，焦点导航会自动跳过它。选择自定义 UTC 时会以当前 UTC 初始化；其年、月、日、时、分、秒采用下划线分段编辑。调整月份或年份会将日期截断到目标月的合法末日，调整日期、时分秒会自动跨月、跨年。日期遵循完整公历闰年规则：4 年一闰，世纪年仅在能被 400 整除时为闰年。JD 输入直接解释为 TT 儒略日，适合可复现的天文计算。

查询输出的时间是按经度与均时差计算的真太阳时；下方同时给出干支、太阳位置、太阳活动时间线和月球位置。纬度和经度支持以下四种形式：

```text
52.5200
52°31′12″
52 31 12
52d31m12s
```

内部统一转换为十进制度。纬度范围为 `0–90°`，经度范围为 `0–180°`，度分秒中的分、秒范围均为 `0–59.999…`；北/南、东/西字段决定最终正负号。

## 计算模型与时间尺度

### 时间

- Unix 时间与 UTC 公历时间。
- JD、MJD、JDN。
- UTC → TT：内置 1972-01-01 起至 2017-01-01 的闰秒表。2017 年后暂按 `TAI−UTC = 37 s` 处理；若未来再出现闰秒，需要更新表。
- TT → UTC：用于将节气天文瞬间显示为 UTC。

UTC 早于 1972 年的 TAI−UTC 关系不属于整秒闰秒表范围，因此当前明确不支持该转换。

### 真太阳时

[solar_time.cpp](solar_time.cpp) 实现视地方时（常称“真太阳时”）：

```text
真太阳时 = UTC + 经度 × 4 分钟 + 均时差
```

东经为正、西经为负。该时间不依赖法定时区，适合作为后续任意坐标查询的时间与干支基准；BER/BEI 预置地点页面仍显示各自的民用时区时间。

### 太阳

[solar_model.cpp](solar_model.cpp) 使用由 [IMCCE VSOP87D Earth](https://ftp.imcce.fr/pub/ephem/planets/vsop87/VSOP87D.ear) 数据机械转换的本地 2,425 项地球 `L/B/R` 级数表 [vsop87d_earth.inc](vsop87d_earth.inc)。计算流程为：

1. 以 TT 儒略日计算地球在日期分点黄道坐标中的日心经度、纬度与距离；
2. 反向得到太阳地心几何位置；
3. 应用主项章动与年周光行差，得到太阳视黄经；
4. 转换得到赤经和赤纬。

均时差目前使用低阶解析式，仅作显示，不参与节气或历法判断。

### 节气

二十四节气由太阳视黄经的方程直接定义：

```text
λ☉ = 15° × n,  n = 0..23
```

[solar_terms.cpp](solar_terms.cpp) 在目标附近建立十天区间，并以二分法求根。程序显示当前节气、下一节气和本年八个主节气：立春、春分、立夏、夏至、立秋、秋分、立冬、冬至。节气瞬间首先以 TT 得到，再转换为 UTC 显示。

### 月球

[moon_model.cpp](moon_model.cpp) 当前是本地周期展开的首版，输出月球地心黄经、黄纬、距离、赤纬，以及有符号日月黄经差、绝对角距和照明比例。

月相方向由有符号日月黄经差判断：正值为渐盈，负值为渐亏。天文数据页的月相盘使用同一受光方向，不使用 emoji 或位图。

该月球模型是完整 ELP82B 接入前的接口与功能基础；尚不应将其用于高精度朔望、食或农历定朔。完整 ELP2000-82B 数据可参考 [IMCCE ELP82B](https://ftp.imcce.fr/pub/ephem/moon/elp82b/)。

## 历法与干支规则

### 日柱与时柱

- 日柱以各地点自身的本地公历日期转换为 JDN 后计算。
- 时支按两个小时为一个时辰计算；时间行额外显示“子初一刻”等刻位。
- 时干由日干与时支计算。

因此 BER 与 BEI 在本地跨日或跨时辰时可得到不同的日柱、时柱；它们不共享北京的历法结果。

### 八字年柱

采用四柱常用的“立春瞬间换年”规则，不采用农历正月初一换年。

立春定义为太阳视黄经 `λ☉ = 315°`。为了避免为每次年柱显示重复求根，程序使用等价的直接判断：一月恒在立春前；二月仅在 `λ☉ < 315°` 时仍属上一八字年。该化简不改变传统规则。

### 八字月柱

八字月柱不按农历月，也不按中气，而按十二个“节”切换。传统“甲己之年丙作首”等口诀在程序中被写为等价数学式：

```text
m = floor(mod(λ☉ − 315°, 360°) / 30°)
月支索引 = (2 + m) mod 12
月干索引 = (2 + 2 × (年干索引 mod 5) + m) mod 10
```

`m = 0` 为寅月，`m = 11` 为丑月。这样只需当前太阳视黄经即可计算月柱；节气求根仍用于显示节气发生时刻。

## 精度与边界

- 节气求根的数值误差很小，但总体精度由太阳模型、章动近似和 UTC↔TT 数据决定。
- 未来 UTC 标记受到未来闰秒不可预知的限制；TT 中天文计算本身不受这一限制。
- 月球首版适合 TUI 展示月相与基础位置，不应作为农历定朔、月食或日食的精确依据。
- 历史农历和民俗纪年存在地区、时区与规则版本差异；不能由一套现代规则自动覆盖。
